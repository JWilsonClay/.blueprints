#### CONTEXT.md
**Purpose and Rationale**: Context provides the foundational knowledge or scenario for the LLM to operate within, reducing reliance on its parametric memory and minimizing factual errors. From a cognitive angle, this mimics human priming; in LLM terms, it leverages the transformer's attention mechanism to prioritize relevant tokens. Nuances include token limits—overly verbose context can dilute focus— and relevance filtering to avoid noise. Implications: In tasks like summarization, insufficient context leads to incomplete outputs; in creative tasks, rich context sparks innovation. Edge cases: Dynamic contexts (e.g., from user history) require adaptive updates.

**Key Components**:
- Background narrative.
- Domain-specific details.
- User-specific info (e.g., preferences).

**Example in Prompt Use**: "You are assisting a beginner programmer in Python. The project involves building a web scraper for news sites, focusing on ethical data collection."

#### CONSTRAINTS.md
**Purpose and Rationale**: Constraints act as guardrails to prevent divergence, ensuring outputs are safe, relevant, and efficient. Viewed through a systems engineering lens, they define the solution space; in LLMs, they counteract tendencies toward verbosity or irrelevance. Multiple angles: Legal/ethical (e.g., no harmful content), performance (e.g., length limits), and quality (e.g., factual accuracy). Nuances: Constraints can be implicit (via examples) or explicit; over-constraining may cause underperformance in open-ended tasks. Implications: In enterprise settings, they enforce compliance; in research, they enable controlled experiments. Edge cases: Conflicting constraints (e.g., "be creative but strictly factual") require prioritization.

**Key Components**:
